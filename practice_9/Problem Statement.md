## Problem Statement

**Система бронирования переговорных комнат (повышенный балл)**

Сотрудники компании нуждаются в удобном инструменте для бронирования переговорных комнат с автоматическими уведомлениями о статусе бронирования и напоминаниями. Система предоставляет веб-интерфейс (Flask + Bootstrap) и REST API для просмотра доступных комнат, создания, изменения и отмены бронирований с автоматической проверкой конфликтов по времени.

**Пользователи:** сотрудники и администраторы.

**Внешние системы:** SMTP-сервер для email-уведомлений, Telegram Bot API для уведомлений через мессенджер.

**Архитектура (микросервисы, повышенный балл):**

- **booking-api** (Flask + SQLAlchemy) — API Gateway, бизнес-логика бронирования, проверка конфликтов
- **notification-worker** (asyncio + aiosmtplib + aiogram + pika) — асинхронная отправка email и Telegram-уведомлений
- **frontend** (Flask + Bootstrap) — веб-интерфейс для пользователей и администраторов
- **PostgreSQL** (StatefulSet + PVC) — основное хранилище данных (комнаты, бронирования, пользователи)
- **RabbitMQ** — очередь сообщений для асинхронного взаимодействия между booking-api и notification-worker
- **SMTP** — внешний сервис для отправки email (поддержка Яндекс, Mail.ru, Gmail, МИРЭА)
- **Telegram Bot API** — внешний сервис для отправки уведомлений (библиотека aiogram)

**Взаимодействие между сервисами:**

1. Пользователь создаёт бронирование через frontend или API → booking-api
2. booking-api проверяет конфликты, сохраняет в PostgreSQL
3. booking-api публикует сообщение в RabbitMQ (очередь `notifications`)
4. notification-worker получает сообщение из очереди
5. notification-worker асинхронно отправляет email (через SMTP) и Telegram (через aiogram)
6. Пользователь получает уведомление о создании бронирования

**Технические требования:**

- Каждый микросервис упакован в Docker-образ
- Docker Compose для локальной разработки
- Развёртывание в Minikube (Deployment, Service, Ingress, StatefulSet, ConfigMap, Secret)
- Service Mesh: Linkerd
- Распределённая трассировка: Jaeger (OpenTelemetry)
- Централизованный сбор логов: Loki + Promtail
- Мониторинг: Prometheus + Grafana (кастомные метрики приложения)

**Минимальные требования (уже выполнены):**
- 3 диаграммы в нотации C4
- Микросервисная архитектура (3+ сервиса)
- Docker-образы для каждого сервиса
- Внешнее хранилище (PostgreSQL)
- Интеграция с внешним сервисом (Telegram Bot API, SMTP)
- Автотесты (5 сценариев)
- Развёртывание в Minikube
- Система мониторинга

**Ключевые функции:**
- Просмотр комнат с фильтрацией по вместимости и оснащению
- CRUD-операции над бронированиями с проверкой пересечений
- Асинхронная отправка уведомлений (email + Telegram) через RabbitMQ
- Панель администратора для управления комнатами и просмотра статистики
- REST API для внешних интеграций
- Отмена бронирования с автоматическим уведомлением
